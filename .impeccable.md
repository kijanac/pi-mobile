# Pico — Design Context

## Design Context

### Users
Developers who run the pi coding agent on their own infrastructure and want to
drive it from their phone. Entry points: TestFlight beta testers and, later,
public users discovering Pico cold — they know what a VPS and Tailscale are,
but not what "the bridge" is. Used on iPhones, often one-handed, frequently at
night or away from a desk. A secondary first-run case: someone joining a
bridge that already exists (a friend's or their own second device).

### Brand Personality
Terminal precision, quiet confidence, engineered warmth. The app speaks in
lowercase, wastes no words, and never decorates without meaning. The logo is
Feynman-diagram inspired: straight, geometric, diagrammatic linework —
professional and artistic, never cartoonish or loose.

### Aesthetic Direction
Dark only (developers, evenings, OLED). Near-black `#0a0a0a` ground, lime
accent `#d4ff3a` used sparingly (10% rule — status, progress, the one action
that matters). JetBrains Mono for code and data; UI text via the existing
`type-*` scale. Hairline borders, surface tints over shadows, flat hierarchy.
Motion: transform/opacity only, fast-out easing (`cubic-bezier(0.22,1,0.36,1)`),
matching the edge-swipe/nav-transition language. Anti-references: glassy
gradients, glowing neon dashboards, cartoonish onboarding illustrations,
marketing-speak.

### Design Principles
1. Truthful states — never show an error for a thing the user hasn't set up
   yet; never promise an action that can't work.
2. The accent is information — lime means "alive/ready/act here," nothing else.
3. Teach in place — empty states and first-run moments explain the system
   (phone → tailscale → your box) instead of pointing at buried settings.
4. Respect the expert — short words, real terms (cloud-init, tailnet, cwd),
   no patronizing tours.
5. One screen, one job — each onboarding step does a single thing and shows
   its own progress.
